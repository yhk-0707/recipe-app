# 学習の進め方について（たたき台）

約1ヶ月で、Web/AWSの知識に加えて Next.js / Laravel / ECS に触れることを想定します。
以下は選択肢と比較材料を用意しました。  
実際の進め方はお任せします。順番の入れ替え、記載外の技術を試すも自由です。

## 1. 現状

- Python・Docker・GitHubはある程度経験あり
- AWSはまだ触り程度
- Web系（TypeScript / React / Next.js / PHP / Laravelなど）はこれから

## 2. 前提

- 静的Webアプリは作成済みという前提です
- CRUDアプリにおける「状態が変更したら再描画する」という作りはReactの考え方の土台です
  - ここまでの学びは必ず活きます

## 3. 全体像

```text
React（軽く） → Next.js
Docker（経験済） → ECS
Laravel（他と依存関係なし・好きなタイミングで実施OK）
AWSに静的サイトデプロイ（好きなタイミングで実施OK）
```

- Next.jsはReactの上に成り立つ技術なので、Reactに軽く触れてから進むのがおすすめ
- ECSはDockerの上に成り立つ技術
  - つまづくのはECSよりAWSそのものかフロント / バックの分離になるはず
- Laravelは既存スタックと依存関係がないので、好きなタイミングの実施で大丈夫

## 4. 選択肢の提案

### 4-1. まずどの項目から着手するか

React/Next.js・AWS・Docker/ECS・Laravelのうち、どれから手をつけるかの比較です。

| 着手する項目                | メリット                                       | デメリット                                                                    |
| --------------------------- | ---------------------------------------------- | ----------------------------------------------------------------------------- |
| React/Next.js系から         | Web知見取得の入口に適している                  | Webデプロイの経験が遅れる                                                     |
| AWSに静的サイトデプロイから | ハンズオンがそのまま使える                     | 開発よりインフラ知見である                                                    |
| Laravelから                 | 他と依存関係がなく、バックエンドの知見も活きる | 言語がPHPに切り替わり実装速度も早くない言語のため、最も切り替えコストが大きい |

おすすめ: React/Next.js系から。  
今やる理由があるためです。  
静的Webアプリの延長線上で、個人学習のハードルが高くないです。  
単純に興味もこちらかなと思いました。  
AWSは少し経験もあるためNext.jsの後でも良いです。  
Laravelは依存関係がなくどのタイミングでも支障がありません。  
Next.js版アプリができて静的Webアプリがデプロイできたら、ECSやLaravelに挑戦すると学びのハードルが低く実装イメージも湧きやすいです。

### 4-2. Reactの学び方：段階的 or いきなりNext.js

| 進め方                        | メリット                                                          | デメリット                                           |
| ----------------------------- | ----------------------------------------------------------------- | ---------------------------------------------------- |
| React基礎 → Next.js（段階的） | Reactの考え方をじっくり理解できる／詰まった時に原因を特定しやすい | React単体に時間がかかる分、Next.jsに使える時間が減る |
| いきなりNext.js               | React未経験でも進められる                                         | ReactとNext.jsの境界が曖昧になりやすい               |

おすすめ: いきなりNext.js寄り。  
最初の1〜2日だけReactで既存サイトのリプレース、小さなコンポーネントを数個書いてから進むと、Next.jsとの境界線を補えます。

### 4-3. 既存静的サイトを使うか、新しい教材を使うか

| 進め方                           | メリット                                                 | デメリット                                                           |
| -------------------------------- | -------------------------------------------------------- | -------------------------------------------------------------------- |
| 既存静的サイトをベースに作り直す | 仕様を知っているため「何が違うのか」を具体的に体感できる | 実装済みのものではあるため、学びの幅が狭くなる懸念                   |
| 新しい教材・チュートリアルで学ぶ | 多くの人と共通の知識を得られる                           | 既存静的サイトの知見から発展させられないため、仕様理解に時間がかかる |

おすすめ: 既存静的サイトをベースに作り直す。  
Next.jsからでも良いので、1,2機能追加できると理想です。

### 4-4. React版でどこまでやるか：複製のみ or 機能追加

| 進め方            | メリット                     | デメリット                   |
| ----------------- | ---------------------------- | ---------------------------- |
| 複製するのみ      | ゴールが明確                 | 基本的な使い方しか触れない   |
| 1つ自分で追加する | より実践的な書き方が身につく | 追加機能の設計に時間がかかる |

おすすめ: どちらでもOK。  
ただしReactかNext.jsのどちらかでは機能追加したいですね。

### 4-6. Next.js版でどこまでやるか：単純移植 or サーバー連携

| 進め方                                             | メリット              | デメリット                    |
| -------------------------------------------------- | --------------------- | ----------------------------- |
| Reactで作ったものをそのまま移植するだけ            | 作業量が少ない        | Next.js独自機能はほぼ触らない |
| データの保存先をlocalStorageからサーバー側に変える | Next.js独自知見がつく | 実装の難易度が上がる          |

おすすめ: サーバー連携。  
少し重いので移植だけしてAWSデプロイ、その後にLaravel触りつつサーバー連携も良いです。
