# Next.js を ECS に載せる手順の意味

これは、`nextjs-learning` を Docker 化して ECS/Fargate + ALB で動かすときに、なぜ各手順が必要なのかを整理したメモ。

## 全体像

この手順は、Next.js アプリを「ECS が期待する形」にそろえるためのもの。
単に動けばよいのではなく、次の3つを同時に満たす必要がある。

- コンテナ内で安定して起動できる
- イメージが必要以上に重くならない
- ALB と RDS を分離して運用できる

## 1. `next.config.ts` に `output: 'standalone'` を入れる理由

`standalone` 出力にすると、Next.js が実行に必要な最小限のファイルを `.next/standalone` にまとめてくれる。
これがないと、実行用イメージに `node_modules` を丸ごと抱え込みやすい。

つまり、これは **軽い本番イメージを作るための前提**。

## 2. Dockerfile が必要な理由

ECS はコンテナを動かすサービスなので、まずアプリを Docker イメージにする必要がある。

2 段構成にする理由は、役割を分けるため。

- ビルドステージ
  - 依存関係のインストール
  - Prisma Client の生成
  - Next.js のビルド
- 実行ステージ
  - 本番で必要な最小限だけを置く

こうすると、最終イメージが小さくなり、配布も起動も楽になる。

## 3. `bunx prisma generate` が必要な理由

このアプリは `src/lib/prisma.ts` で `@prisma/client` を使っている。
Prisma Client は生成済みである必要があるので、`next build` の前に `prisma generate` を通しておかないと壊れる。

要するに、**Next.js のビルド前に Prisma のコード生成を済ませる** という順番が必要。

## 4. `.dockerignore` が必要な理由

`COPY . .` をそのまま使うと、不要なものまで全部コンテナに入る。

除外したいものは主にこれ。

- `.env*`
  - 秘密情報がイメージに混ざるのを防ぐ
- `node_modules`
  - ビルドコンテキストを無駄に重くしない
- `.next`
  - ローカルの古い成果物を持ち込まない
- `.git`
  - 履歴や不要なデータを送らない

これは安全性と速度の両方に効く。

## 5. `/api/health` が必要な理由

ALB はタスクの生死判定にヘルスチェックを使う。
ここで DB を見に行くと、DB 障害のせいでアプリ本体まで落ちた扱いになりかねない。

だからヘルスチェックは DB に触らない軽いエンドポイントにする。
`/api/health` はそのためのもの。

## 6. ローカルで `docker run` を確認する理由

ECS に上げる前に、まずコンテナ内で正しく起動できるかを確かめる。

ここで確認したいのは主にこの2つ。

- コンテナ起動時にアプリが落ちないか
- ブラウザ表示と `/api/health` が通るか

ローカルの DB を使って確認するのは、ECS デプロイ前にアプリ側の問題を切り分けるため。

## 7. AWS CLI v2 のセットアップが必要な理由

次の工程では ECR にイメージを push して、ECS にデプロイする。
そのため AWS に認証できる状態が必要になる。

`aws configure` と `aws sts get-caller-identity` を先に通しておくのは、その準備。

## 結論

この手順は、単に「Docker 化する」ためではなく、**Next.js を ECS で無理なく運用できる形へ整えるための準備**。

Next.js、Docker、Prisma、ECR、ECS、ALB の役割分担が一通りつながる。
