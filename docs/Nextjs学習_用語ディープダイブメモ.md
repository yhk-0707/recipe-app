# Next.js学習にあたって - 用語ディープダイブメモ

## レンダリングについて
レンダリングとは、コンポーネントの中身（JSXやstate）を、実際にブラウザが表示できるHTML/DOMに変換する作業。  
ブラウザが、コードを解析して、『画面』に変換して映し出すこと。画面の組み立て。

## レンダリング手法について

### 1. CSR (Client-Side Rendering)
HTMLを組み立てる作業をブラウザの中でやる方式。
従来のReactアプリ（Create React Appなど）の基本スタイルです。  
サーバーからは「中身が空っぽのHTML」と「JavaScriptの塊」だけが送られてきます。  
ブラウザに届いてから、JavaScriptが動いて画面を組み立てます。

- HTMLを作る場所： ブラウザ（クライアント側）
- 作るタイミング： ユーザーがページにアクセスした「後」

- メリット： 一度ページを読み込めば、その後のページ遷移が爆速（サクサク動く）。
- デメリット： 最初の読み込み時に白い画面が続く時間がある。Googleなどのロボットが中身を読み取りにくいため、SEO（検索エンジン最適化）に不利になることがある。

- おすすめのユースケース：ログイン後の管理画面・ダッシュボードなど「検索エンジンに拾われる必要がなく、開く人がログイン済みで限られる」画面。（SEOが不要で、ログイン後しか見ないページ）

---

### 2. SSR (Server-Side Rendering)
HTMLを組み立てる作業を、リクエストが来るたびにサーバー側でやる方式。
ユーザーから「このページ見せて！」とリクエストが来るたびに、サーバー側で最新データをAPIなどから取得し、HTMLを組み立ててからブラウザに返します。

- HTMLを作る場所： サーバー
- 作るタイミング： ユーザーがページにアクセスした「その瞬間（リクエストごと）」

- メリット： 常に最新のデータが表示される。最初から文字が入ったHTMLが届くので、SEOに強い。
- デメリット： アクセスされるたびにサーバーが頑張って生成するので、表示までに少し時間がかかる。アクセスが集中するとサーバーに負荷がかかる。

- おすすめのユースケース：「ログインユーザーのマイレシピ」「在庫の残り数」など、アクセスするたびに中身が変わるページ。（常に最新、かつユーザーごとに内容が違う）

---

### 3. SSG (Static Site Generation)
HTMLをデプロイする前に1回だけ作り置きしておく方式。
Next.jsで最も推奨される基本形です。アプリをデプロイ（公開）するタイミングで、全てのページのHTMLをあらかじめ作ってしまいます。  
ユーザーがアクセスしたときは、すでに完成しているHTMLをただ配るだけです。

- HTMLを作る場所： サーバー（ビルド時）
- 作るタイミング： アプリを本番環境に公開（ビルド）した「とき」

- メリット： すでに完成品を渡すだけなので、圧倒的に表示が速い。サーバーの負荷も最小限。SEOも最強。
- デメリット： データの更新（ブログの修正など）があっても、アプリ全体をもう一度ビルドし直さないと画面に反映されない。リアルタイムなデータ表示には不向き。

- おすすめのユースケース：公開レシピの詳細ページのように、誰が見ても同じで、更新頻度が低いページ。（誰が見ても同じ内容）

---

### 4. ISR (Incremental Static Regeneration)
SSGの「データの更新が大変」という弱点を克服した、Next.jsの強力な機能です。  
「このページは10秒ごとに新しくしてOK」と設定しておくと、SSGの速さを保ちつつ、裏側で定期的に最新のHTMLにアップデートしてくれます。

- HTMLを作る場所： サーバー
- 作るタイミング： 基本はSSG（ビルド時）＋ 指定した時間が経ったあとのアクセス時

- メリット： SSG並みに爆速で表示されつつ、データも（タイムラグはあれど）自動で最新に更新される。
- デメリット： 設定した時間（例: 60秒）の間は、古いデータが表示され続ける可能性がある。完璧なリアルタイム性が必要な画面には使えない。

- おすすめのユースケース：SSGの作り置きHTMLを、決めた時間ごとに裏側で作り直す方式。（頻繁に更新されるが、数分のズレは許容できる）

---

→Reactはユーザーのブラウザ側でレンダリングをする(CSR)。  
　Next.jsはプラスして、ブラウザでサーバー側でレンダリングをすることもできる(SSR,SSG(ISR))。

---

## ハイドレーションについて

サーバーが作った「見た目だけのHTML」に、ブラウザ側のReactが後からクリックやuseStateなどの「命」を吹き込む処理。  
hydrate = 「水を与える」。 つまり、乾いたHTMLにJSという水を注いで動かす、というイメージ。  
SSR・SSG・ISRの方式で必要になる。  

→ 表示はHTMLが先、JSXは後。

### CSRでは不要？

最初からブラウザ上でJavaScriptを使ってHTMLをゼロから組み立てるので、最初から動く状態で作られます。だからハイドレーションというステップはありません。

## ストリーミングについて
非同期処理によるデータ取得をアプリケーションに組み込むための機能。
ページの中で遅いデータ取得がある部分だけを後回しにし、準備できた部分から先にHTMLを届ける仕組み。  
「全部揃うまで真っ白」を防ぎます。  
→そこで使うのが`<Suspense>`タグ。囲んだ部分を非同期コンポーネント化

## RSCペイロードについて
React Server Component Payload  
→ javaScriptでもhtmlでもなく、JSONを強化したような特殊なテキストデータ。

サーバーコンポーネントは、初回はHTMLとして届きますが、ページ内遷移のときは特殊な軽量データ形式（RSCペイロード）としてやり取りされます。

ブラウザに届いたあと、ReactはHTMLの見た目に対して、RSCペイロードのデータを見ながら、JSXを正しくハイドレーションさせます。

### 中身は
- サーバーコンポーネント（RSC）がレンダリングした結果
- データベースから取ってきた生のデータ
- クライアントコンポーネント（use client）をどこにはめ込むかという配置指示書と、そこに渡すデータ（props）

1回目（初回）：「HTML」という大きなファイルの中に、画面の見た目も、RSCペイロードも全部コミコミで1回で届く。

2回目（ページ遷移）：もうHTMLの箱は不要なので、純粋な「RSCペイロードのテキスト」だけが、通信の波に乗ってシュッと届く。

---

※ use clientとは？  
Next.jsでは「サーバーコンポーネント」と、「クライアントコンポーネント」の2つがある。  
→Reactで従来使用していたフックは「クライアントコンポーネント」として扱われる。  
→→Next.jsではデフォルトが「サーバーコンポーネント」となるため、`"use client"`という宣言が必要になる。

Next.js（App Router）の設計の基本は、「基本はすべてサーバーコンポーネント（RSC）で作り、ユーザーが操作する『動く部品』だけをクライアントコンポーネント（RCC）にする」という形。
